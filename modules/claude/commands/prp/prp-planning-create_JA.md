# 計画PRPの作成（高度）

大まかなアイデアを豊富なビジュアルドキュメントを含む包括的なPRDに変換します。

## アイデア: $ARGUMENTS

## 発見プロセス

1. **コンセプト拡張**
   - コアアイデアを分解
   - 成功基準を定義
   - 提供されている場合はビジネス目標にマッピング

2. **市場と技術リサーチ**
   - 以下のための深いWeb検索を実行:
     - 市場分析
     - 競合分析
     - 技術的実現可能性調査
     - ベストプラクティスの例
     - 統合の可能性

3. **ユーザーリサーチと明確化**
     - 提供されていない場合、以下についてユーザーに質問:
     - ターゲットユーザーペルソナ？
     - 主要な課題？
     - 成功指標？
     - 制約/要件？

## PRD生成

`/PRPs/templates/prp_planning_base.md`を使用:

### ビジュアルドキュメント計画
```yaml
diagrams_needed:
  user_flows:
    - ハッピーパスジャーニー
    - エラーシナリオ
    - エッジケース

  architecture:
    - システムコンポーネント
    - データフロー
    - 統合ポイント

  sequences:
    - APIインタラクション
    - イベントフロー
    - 状態変更

  data_models:
    - エンティティリレーションシップ
    - スキーマ設計
    - ステートマシン
```

### リサーチ統合
- **市場分析**: PRDに調査結果を含める
- **技術オプション**: アプローチを比較
- **リスク評価**: 軽減戦略付き
- **成功指標**: 具体的で測定可能

### ユーザーストーリー開発
```markdown
## エピック: [高レベル機能]

### ストーリー1: [ユーザーニーズ]
**[ユーザータイプ]として**
**[機能]が欲しい**
**[利益]のために**

**受け入れ基準:**
- [ ] 具体的な動作
- [ ] エッジケース処理
- [ ] パフォーマンス要件

**技術ノート:**
- 実装アプローチ
- API影響
- データ要件
```

### 実装戦略
- 依存関係のあるフェーズ（日付なし）
- 優先順位付け
- MVPと拡張機能
- 技術的前提条件

## ユーザーインタラクションポイント

1. **アイデア検証**
   - 理解を確認
   - 曖昧さを明確化
   - 境界を設定

2. **リサーチレビュー**
   - 調査結果を共有
   - 仮定を検証
   - 方向を調整

3. **PRDドラフトレビュー**
   - アーキテクチャ承認
   - リスク認識
   - 成功指標合意

## 図表ガイドライン
- すべての図表にMermaidを使用
- 必要に応じて凡例を含める
- エラーパスを表示
- 複雑なフローに注釈を付ける

## 出力構造
```markdown
1. エグゼクティブサマリー
2. 問題とソリューション
3. ユーザーストーリー（図表付き）
4. 技術アーキテクチャ（図表付き）
5. API仕様
6. データモデル
7. 実装フェーズ
8. リスクと軽減策
9. 成功指標
10. 付録
```

保存先: `PRPs/{feature-name}-prd.md`

## 品質チェックリスト
- [ ] 問題が明確に説明されている
- [ ] ソリューションが問題に対処している
- [ ] すべてのユーザーフローが図表化されている
- [ ] 必要に応じてワイヤーフレームを含む
- [ ] アーキテクチャが視覚化されている
- [ ] APIが例付きで完全に仕様化されている
- [ ] データモデルが含まれている
- [ ] 依存関係が特定されている
- [ ] リスクが特定され軽減策がある
- [ ] 成功指標が測定可能
- [ ] 実装フェーズが論理的
- [ ] 実装PRP準備完了

覚えておいてください: 優れたPRDは実装の混乱を防ぎます。
