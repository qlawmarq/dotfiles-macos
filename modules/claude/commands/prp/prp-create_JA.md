# ベースPRPの作成

## 機能: $ARGUMENTS

## PRP作成ミッション

**ワンパス実装成功**を可能にする、体系的なリサーチとコンテキストキュレーションを通じた包括的なPRPを作成します。

**重要な理解**: 実行AIエージェントが受け取るのは:

- まず、PRPs/README.mdを読んでPRP概念を理解してください
- あなたが作成するPRPの内容
- トレーニングデータの知識
- コードベースファイルへのアクセス（ただし、どのファイルかのガイダンスが必要）

**したがって**: あなたのリサーチとコンテキストキュレーションが実装の成功を直接決定します。不完全なコンテキスト = 実装の失敗。

## リサーチプロセス

> リサーチプロセス中に、明確なタスクを作成し、バッチツールを使用して必要な数のエージェントとサブエージェントを生成します。ここで行う深いリサーチがあるほど、PRPは良くなります。成功の可能性を最適化し、速度ではなく最適化します。

1. **コードベース分析を深く**
   - 明確なタスクを作成し、コードベースで類似機能/パターンを検索するサブエージェントを生成。アプローチをよく考えて計画
   - PRPで参照する必要のあるすべてのファイルを特定
   - 従うべき既存の規約をすべて記録
   - 検証アプローチのため既存のテストパターンを確認
   - バッチツールを使用して、コードベースで類似機能/パターンを検索するサブエージェントを生成

2. **大規模な外部リサーチ**
   - 類似機能/パターンをオンラインで深くリサーチするサブエージェントを作成し、指示を与える。ドキュメントや例へのURLを含める
   - ライブラリドキュメント（特定のURLを含める）
   - 重要なドキュメント部分については、PRPs/ai_docsに.mdファイルを追加し、PRPで参照。明確な理由と指示を記述
   - 実装例（GitHub/StackOverflow/ブログ）
   - リサーチ中に見つけたベストプラクティスと一般的な落とし穴
   - バッチツールを使用して、類似機能/パターンをオンラインで検索し、ドキュメントや例へのURLを含めるサブエージェントを生成

3. **ユーザーへの明確化**
   - 必要に応じて明確化を求める

## PRP生成プロセス

### ステップ1: テンプレートを選択

テンプレート構造として`PRPs/templates/prp_base.md`を使用 - 必要なすべてのセクションとフォーマットが含まれています。

### ステップ2: コンテキスト完全性の検証

書く前に、テンプレートから**「事前知識なし」テスト**を適用:
_「このコードベースについて何も知らない人が、これを成功裏に実装するために必要なすべてを持っているか？」_

### ステップ3: リサーチ統合

リサーチ結果をテンプレートセクションに変換:

**目標セクション**: リサーチを使用して、特定の測定可能な機能目標と具体的な成果物を定義
**コンテキストセクション**: リサーチ結果でYAML構造を埋める - 特定のURL、ファイルパターン、注意点
**実装タスク**: コードベース分析からの情報密度の高いキーワードを使用して、依存関係順のタスクを作成
**検証ゲート**: このコードベースで動作することを確認したプロジェクト固有の検証コマンドを使用

### ステップ4: 情報密度基準

すべての参照が**具体的で実行可能**であることを確認:

- URLにはセクションアンカーを含め、ドメイン名だけでなく
- ファイル参照には従うべき特定のパターンを含め、一般的な言及ではなく
- タスク仕様には正確な命名規則と配置を含める
- 検証コマンドはプロジェクト固有で実行可能

### ステップ5: 書く前にULTRATHINK

リサーチ完了後、TodoWriteツールを使用して包括的なPRP作成計画を作成:

- リサーチ結果で各テンプレートセクションを構成する方法を計画
- 追加リサーチが必要なギャップを特定
- 実行可能なコンテキストでテンプレートを埋めるための体系的なアプローチを作成

## 出力

保存先: `PRPs/{feature-name}.md`

## PRP品質ゲート

### コンテキスト完全性チェック

- [ ] テンプレートからの「事前知識なし」テストに合格
- [ ] すべてのYAML参照が具体的でアクセス可能
- [ ] 実装タスクに正確な命名と配置ガイダンスを含む
- [ ] 検証コマンドがプロジェクト固有で動作確認済み

### テンプレート構造準拠

- [ ] すべての必須テンプレートセクションが完成
- [ ] 目標セクションに特定の機能目標、成果物、成功定義がある
- [ ] 実装タスクが依存関係順に従う
- [ ] 最終検証チェックリストが包括的

### 情報密度基準

- [ ] 一般的な参照なし - すべてが具体的で実行可能
- [ ] ファイルパターンが従うべき特定の例を指す
- [ ] URLに正確なガイダンスのためのセクションアンカーを含む
- [ ] タスク仕様がコードベースからの情報密度の高いキーワードを使用

## 成功指標

**信頼スコア**: ワンパス実装成功の可能性を1-10で評価

**検証**: 完成したPRPは、コードベースに不慣れなAIエージェントが、PRPの内容とコードベースアクセスのみを使用して機能を成功裏に実装できるようにする必要があります。
